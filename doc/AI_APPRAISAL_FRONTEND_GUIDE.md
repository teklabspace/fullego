# AI Appraisal & AI Asset Review — Frontend Integration Guide

This document describes the AI features added to the Assets module:

1. **Automated Appraisal** — an instant, AI-generated value estimate for an asset.
2. **Concierge Service** — a human-expert appraisal (existing flow, unchanged).
3. **AI Asset Review** — an advisory AI accept/reject decision based on the asset's
   details and uploaded documents.
4. **AI Usage** — per-plan monthly quotas the frontend should surface to the user.

All endpoints are under the Assets router.

- **Base URL:** `{API_BASE}/api/v1/assets`
- **Auth:** every request needs `Authorization: Bearer <access_token>` (the same token used for all other asset endpoints).
- **Content-Type:** `application/json`

---

## 1. Concept: Automated vs Concierge

| | **Automated Appraisal** | **Concierge Service** |
|---|---|---|
| Engine | AI (Claude) | Human expert |
| Speed | Instant (synchronous) | 3–5 business days |
| Output | Estimated value + range + reasoning | Full report + recommendations |
| Status after request | `completed` immediately | `pending` (handled by staff) |
| Certified? | **No** — estimate only, always show the disclaimer | Yes (professional) |
| API trigger | `POST /{id}/appraisals` with `appraisal_type: "API"` | `POST /{id}/appraisals` with `appraisal_type: "Concierge"` |

**Important UX rule:** for an Automated Appraisal you **must** display the returned
`disclaimer` text near the value. It is an estimate, not a guaranteed sale price or
certified appraisal.

## 2. Automated Appraisal

### Request

`POST /api/v1/assets/{asset_id}/appraisals`

```json
{
  "appraisal_type": "API",
  "notes": "optional",
  "preferred_date": null
}
```

- `appraisal_type` is **case-insensitive**. `"API"`, `"api"` both work. Use `"API"` for
  the Automated (AI) appraisal.
- `notes` and `preferred_date` are ignored for the Automated path (they apply to Concierge).

### Success response — `201 Created`

```json
{
  "data": {
    "id": "f1c2…",
    "asset_id": "a9b8…",
    "appraisal_type": "API",
    "status": "completed",
    "requested_at": "2026-06-23T14:00:00Z",
    "estimated_completion_date": null,
    "estimated_cost": null,
    "completed_at": "2026-06-23T14:00:03Z",
    "report_url": null,
    "estimated_value": 12000.00,
    "notes": "Comparable luxury watches in this condition sell in this band."
  },
  "ai_result": {
    "estimated_value": 12000.00,
    "value_range_low": 10000.00,
    "value_range_high": 14000.00,
    "currency": "USD",
    "confidence": "medium",
    "reasoning": "Comparable luxury watches in this condition sell in this band.",
    "disclaimer": "Estimated value generated by the platform based on available data. This is not a certified professional appraisal or a guaranteed sale price.",
    "model": "claude-opus-4-8"
  }
}
```

- Read the rich AI fields from **`ai_result`** (value, range, confidence, reasoning, disclaimer).
- `data` is the persisted appraisal record (same shape as the existing appraisal list).
- Side effects (already applied by the backend): the asset's `estimated_value` is updated,
  its `valuation_type` becomes `"automated"`, and a valuation-history entry is created — so a
  refresh of the asset / value-history views will show the new estimate.

### Suggested UI

- Show `estimated_value` large, with `value_range_low`–`value_range_high` as a sub-line
  ("Range: $10,000 – $14,000").
- Show a `confidence` badge: `low` / `medium` / `high`.
- Show `reasoning` in an expandable section.
- Always show `disclaimer` as muted helper text under the value.

## 3. Concierge Appraisal (existing, unchanged)

`POST /api/v1/assets/{asset_id}/appraisals`

```json
{
  "appraisal_type": "Concierge",
  "notes": "High-value watch, need insurance-grade report",
  "preferred_date": "2026-07-01T00:00:00Z"
}
```

Response — `201 Created`:

```json
{ "data": { "id": "…", "appraisal_type": "Concierge", "status": "pending", "estimated_value": null, "...": "..." } }
```

No `ai_result` is returned. The request stays `pending` until staff complete it via the
Concierge workflow. List/poll with the existing endpoints:

- `GET /api/v1/assets/{asset_id}/appraisals`
- `GET /api/v1/assets/{asset_id}/appraisals/{appraisal_id}`

## 4. AI Asset Review (accept / reject — advisory)

The AI reads the asset's details and the metadata of its uploaded documents, and returns a
decision. **The decision is advisory:** it is recorded on the asset but does **not** hide or
block the asset. Use it to guide the user (e.g. "looks good" / "needs more documents").

### Run a review

`POST /api/v1/assets/{asset_id}/ai-review`

No request body.

Success — `201 Created`:

```json
{
  "data": {
    "id": "r1a2…",
    "asset_id": "a9b8…",
    "decision": "needs_review",
    "reason": "Certificate is present but the purchase receipt is missing.",
    "flags": ["No purchase receipt", "Value not independently verified"],
    "model": "claude-opus-4-8",
    "created_at": "2026-06-23T14:05:00Z"
  }
}
```

`decision` is one of:

| Value | Meaning | Suggested badge |
|---|---|---|
| `approved` | Coherent details, documents present | Green ✓ |
| `rejected` | Clear inconsistency / invalid data | Red ✕ |
| `needs_review` | Inconclusive — a human should decide | Amber ⚠ |
| `not_reviewed` | (asset default before any review is run) | Grey |

- `flags` is an array of short strings (may be empty). Render as a bullet list.
- The asset object also carries the latest verdict on its own fields:
  `ai_review_status` (same enum) and `ai_reviewed_at` (timestamp) — useful for showing a
  status chip on the asset card without a second call.

### Get the latest review

`GET /api/v1/assets/{asset_id}/ai-review`

```json
{ "data": { "id": "…", "decision": "approved", "reason": "…", "flags": [], "model": "…", "created_at": "…" } }
```

If no review has ever been run, `data` is `null`:

```json
{ "data": null }
```

## 5. AI Usage & Quotas

AI usage is capped per subscription plan, **per calendar month**. Show the remaining quota
so users understand the limit before they hit it.

### Get current usage

`GET /api/v1/assets/ai/usage`

```json
{
  "plan": "free",
  "period": "2026-06",
  "ai_appraisals": { "limit": 3, "used": 1, "remaining": 2 },
  "ai_reviews":    { "limit": 3, "used": 0, "remaining": 3 }
}
```

- `limit: null` and `remaining: null` mean **unlimited** (annual plan).
- `period` is the month the counters apply to (`YYYY-MM`, UTC).

### Default limits

| Plan (`plan`) | `ai_appraisals_per_month` | `ai_reviews_per_month` |
|---|---|---|
| `free` | 3 | 3 |
| `monthly` | 25 | 25 |
| `annual` | unlimited | unlimited |

(Admins are exempt from all AI limits.)

## 6. Error responses

All errors use the standard shape:

```json
{ "detail": "Human-readable message" }
```

| HTTP | When | Frontend handling |
|---|---|---|
| `401` | Missing/invalid token | Redirect to login |
| `403` | Monthly AI limit reached. `detail` e.g. *"Automated appraisal limit reached (3/month for your plan). Upgrade for more."* | Show upgrade prompt; disable the AI button |
| `404` | Asset not found / not owned by the user | Show not-found state |
| `503` | AI not configured or temporarily unavailable (e.g. provider error). `detail` e.g. *"The AI service is temporarily unavailable. Please try again."* | Show retry; do not treat as a permanent failure |
| `422` | Malformed body / bad enum value | Validation error UI |

**Recommended flow for the AI buttons:**
1. On page load, call `GET /assets/ai/usage`.
2. If `remaining === 0` (and `limit !== null`), disable the button and show "Upgrade to run more".
3. On click, call the endpoint; on `403` re-fetch usage and show the upgrade prompt; on `503`
   show a transient "try again" toast.

## 7. Enum quick reference

| Field | Allowed values |
|---|---|
| `appraisal_type` (request) | `API` (Automated/AI), `Concierge` (human), `Standard`, `Comprehensive`, `Expedited`, `Insurance` — case-insensitive |
| appraisal `status` | `pending`, `in_progress`, `completed`, `cancelled` |
| review `decision` / asset `ai_review_status` | `not_reviewed`, `approved`, `rejected`, `needs_review` |
| `confidence` | `low`, `medium`, `high` |
| `plan` | `free`, `monthly`, `annual` |

## 8. Endpoint summary

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/api/v1/assets/{id}/appraisals` (`type=API`) | Run an instant AI valuation |
| `POST` | `/api/v1/assets/{id}/appraisals` (`type=Concierge`) | Request a human appraisal |
| `GET`  | `/api/v1/assets/{id}/appraisals` | List an asset's appraisals |
| `GET`  | `/api/v1/assets/{id}/appraisals/{appraisal_id}` | Get one appraisal |
| `POST` | `/api/v1/assets/{id}/ai-review` | Run an AI accept/reject review |
| `GET`  | `/api/v1/assets/{id}/ai-review` | Get the latest AI review (or `null`) |
| `GET`  | `/api/v1/assets/ai/usage` | Current month AI quota & usage |

## 9. Backend prerequisites (for the team, not the frontend)

These must be done on the server before the AI endpoints work:

1. `pip install -r requirements.txt` (adds the `anthropic` SDK).
2. `alembic upgrade head` (adds the `asset_ai_reviews` table, asset AI columns, and the
   `automated` valuation type).
3. `.env` must contain a valid `ANTHROPIC_API_KEY`. Optional: `AI_MODEL` (default
   `claude-opus-4-8`) and `AI_FEATURES_ENABLED` (default `true`).

If the key is missing or `AI_FEATURES_ENABLED=false`, the AI endpoints return `503` and the
rest of the app is unaffected.
